MITRE ATT&CK Framework

MITRE ATT&CK is a globally recognised knowledge base of adversary tactics, techniques, and procedures (TTPs). Sportetitive's attack maps directly onto multiple ATT&CK Enterprise techniques, each of which suggests a specific defensive countermeasure.

Cyber Kill Chain

Developed by Lockheed Martin, the Cyber Kill Chain describes the seven stages an attacker must progress through to achieve their objective. Disrupting any stage stops or limits the attack. The Sportetitive attack progressed through all seven stages.

Lockheed Martin Kill Chain overview →
1. Reconnaissance
LinkedIn post, public OSINT, no network contact
Break it: Employee OSINT training; limit professional social media exposure; dark web monitoring
2. Weaponisation
Fake intranet site, CEO-spoofed email crafted
Break it: DMARC email authentication; brand impersonation monitoring
3. Delivery
Phishing link clicked by Pete Mitchell; credentials captured
Break it: Anti-phishing training; MFA so captured password alone is insufficient; email gateway filtering
4. Exploitation
MSP access used; keylogger deployed via RMM toolset
Break it: Restrict and monitor MSP toolset permissions; EDR on all endpoints; least-privilege
5. Installation
Persistent access established via captured domain admin credentials
Break it: PAM with just-in-time privilege; alerting on off-hours admin logins; behaviour analytics
6. C2 & Lateral movement
RDP lateral movement; two weeks of undetected exploration
Break it: Network segmentation; SIEM; Zero Trust assume-breach posture; SOC monitoring
7. Actions on Objectives
Data exfiltration then ransomware deployment; ransom demand via CEO email
Break it: DLP; immutable backups; tested incident response plan; IR retainer

Zero Trust Architecture

Zero Trust is a security model built on the principle “never trust, always verify”. It eliminates the assumption that anything inside the network perimeter is safe. The Sportetitive attack succeeded precisely because implicit trust was placed in the MSP, in domain admin credentials, and in internal network traffic. Zero Trust removes every one of these assumptions.

NCSC Zero Trust guidance →

Verify explicitly

Every access request — from any user, device, or service, inside or outside the network — is authenticated, authorised, and continuously validated. MFA on every account is the minimum starting point. No account accesses systems without proof of identity.

Use least privilege

Every user and service account has only the access it needs to do its job. Domain administrator accounts are not used for daily tasks. Just-in-time access is requested and approved for specific tasks with time limits. Pete Mitchell's MSP account had access that far exceeded what was needed.

Assume breach

Design systems as if an attacker is already inside. Segment the network so a compromised MSP account cannot reach customer databases. Monitor all lateral movement. Enforce data-access controls so exfiltration requires multiple steps. The Sportetitive network had no segmentation that would have slowed the attacker.

Verify devices

Only managed, compliant devices can access corporate systems. BYOD and supplier devices without management profiles are not trusted. This would have prevented MSP access from an unmanaged device.

Monitor continuously

Log everything. Alert on anomalies in real time. A SIEM with behaviour-based rules would have flagged off-hours admin logins, unusual lateral movement, and large outbound data transfers long before encryption began. Sportetitive had no 24/7 monitoring.

Protect data explicitly

Classify data by sensitivity. Encrypt at rest and in transit. Apply access controls at the data level, not just the network level. Customer personal data and employee payroll records require their own access control layer — independent of network-level trust.

NIST SP 800-61 — Computer Security Incident Response

The NIST Special Publication 800-61r2 is the US government's incident response lifecycle standard. It is widely adopted in the UK, underpins Sportetitive's incident response plan, and defines the four-phase framework that the 72-hour playbook follows.

NIST SP 800-61r2 (PDF) →
1

Preparation

Building the policies, procedures, tools, training, and relationships needed before an incident occurs. Sportetitive's gap: no tested incident response plan, no cyber insurer retainer, no IR firm on contract.

2

Detection & Analysis

Identifying that an incident has occurred and understanding its scope and severity. Sportetitive's gap: the attacker was inside for 13 days before detection because no 24/7 monitoring or SIEM was in place.

3

Containment, Eradication & Recovery

Isolating affected systems, removing the attacker, restoring from clean backups. Covered in full in the 72-hour playbook. The recovery order prioritises identity, monitoring, data, IMS, and website in sequence.

4

Post-Incident Activity

Lessons-learned review, documentation, evidence preservation, improvement actions. Sportetitive's 10-working-day post-incident review feeds directly into the three-phase improvement plan.
